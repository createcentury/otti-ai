# freee-check Skill — 詳細仕様

## 概要

freee の全顧問先（または指定事業所）の未処理明細を自動チェックし、
仕訳分類・登録まで自律的に行う税務事務所向けコアスキル。

## 処理フロー（詳細）

### 事前フェーズ

```python
# Ralph Loop によって毎晩21時に自動起動
# loop.sh --cron "21:00" --skill freee-check
```

### Phase 1: データ取得と絞り込み

freee MCP を使って以下の条件で明細を取得:

```
フィルター条件:
- status: "未処理"
- type: "出金"
- date: 当期会計年度内
- 取得順: 事業所ID昇順
```

各事業所の処理は `company_id` 単位で完全分離。
別会社のデータが混入した場合は即座に処理を停止。

### Phase 2: 除外ルールの適用（確認ゲート振り分け）

以下の7パターンに該当する明細を除外:

| パターン | キーワード例 |
|---|---|
| 内容不明デビット | 摘要が空、または「デビット」のみ |
| 借入金返済 | ローン, 借入, 返済 |
| 税金・社会保険料 | 国税, 都税, 社会保険, 年金, 労働保険 |
| 給与支払い | 給与, 賞与（※従業員名簿と突合） |
| 投資 | 株式, 投資信託, 証券, ETF |
| ATM出金 | ATM, 現金引き出し |
| 公共料金 | 電力, ガス, 水道, NHK |

除外した明細は `logs/YYYY-MM-DD/[company_id]-exceptions.log` に記録。

### Phase 3: 仕訳分類（2段階判定）

#### Step 3-1: キーワード辞書マッチング

正規化処理（全角→半角、大文字→小文字）後に辞書照合:

```python
# 辞書の構造例
categories = {
    "外注費": ["業務委託", "外注", "フリーランス"],
    "通信費": ["aws", "google cloud", "azure", "github", "notion"],
    "広告宣伝費": ["google ads", "meta", "facebook", "twitter"],
    # ... 14カテゴリ・各100個以上のキーワード
}

# 特殊ルール
special_rules = [
    {
        "condition": "振込 AND カタカナ人名（regex: [ァ-ン]{2,}）",
        "account": "外注費",
        "date_adjustment": "前月末に変更"
    }
]
```

マッチした場合: 仕訳候補として確定（次のPhaseへ）
マッチしなかった場合: Phase 3-2 へ

#### Step 3-2: LLM フォールバック（Claude API）

辞書でマッチしなかった明細のみ Claude API に投げる:

```
プロンプト構造:
- 摘要: [明細テキスト]
- 金額: [金額]
- 取引日: [日付]
- 選択肢: [14科目のリスト]
- 出力形式: JSON（account, confidence: High/Medium/Low, reason）
```

信頼度による分岐:
- High/Medium: 仕訳候補として確定（確認ゲートへ）
- Low: 例外ログへ（人間確認）

### Phase 4: 重複チェックと本登録

既存取引と以下の3条件で突合:
- 取引日が同一
- 金額が同一
- 摘要の先頭40文字が同一

重複なし → 確認ゲートへ提出
重複あり → スキップ（ログに記録）

**確認ゲート**: 分類結果を表形式で表示、承認後に freee MCP で登録

新規取引先の場合: 取引先マスタも自動生成（確認ゲートあり）

### Phase 5: セキュリティ分離とロギング

- CLI 画面には金額・取引先名を出力しない
- 各事業所のログは独立したファイルに書き出す
- マルチテナント汚染を検知したら即停止

### 事後フェーズ: ROI レポート

## エラーハンドリング

| エラー種別 | 対応 |
|---|---|
| freee API 接続エラー | 3回リトライ後、Slack通知して停止 |
| 認証エラー | 即停止、Slack通知 |
| マルチテナント汚染検知 | 即停止、Slack通知、該当ログを保全 |
| 判定不能（LLM信頼度Low） | スキップして例外ログへ |
