# SOP → Skill 変換ガイド

otti-ai の核心機能: 業務手順書（SOP）を Claude Code のスラッシュコマンドに変換する。

---

## なぜ SOP → Skill 変換が重要か

高度なプログラミングスキルは不要。
**業務を論理的に分解できれば**、それをSkillに変換できる。

```
税理士の「仕訳ルール」         → /freee-check
マーケターの「広告作成手順」   → /rsa
営業の「提案書作成フロー」     → /proposal-gen
人事の「採用評価基準」         → /candidate-eval
```

---

## 変換ステップ

### Step 1: SOPを「処理フロー」に分解する

良いSOPの条件:
- 入力が明確（何を受け取るか）
- 出力が明確（何を生成するか）
- ステップが番号付きで列挙されている
- 判断分岐（if/else）が明示されている

**変換前（業務手順書）**:
```
未処理の請求書が届いたら：
1. 金額と取引先を確認する
2. 過去の請求書と照合して重複がないか確認
3. 3万円未満なら自分で承認、以上なら上長に回す
4. 承認後にシステムに入力する
```

**変換後（Skill フォーマット）**:
```markdown
# invoice-check — 請求書の受取・照合・承認フロー

## 引数
$ARGUMENTS（請求書ファイルパスまたは金額・取引先テキスト）

## 実行手順
1. 請求書から金額と取引先名を抽出
2. `logs/invoices/` で過去の請求書と照合（重複チェック）
3. 金額による分岐:
   - 30,000円未満: 承認ゲート（自分）
   - 30,000円以上: 確認ゲート（上長への確認を求める）
4. 承認後にシステムへ入力（確認ゲートあり）
```

### Step 2: /sop-convert コマンドで自動変換

```
claude
> /sop-convert

未処理の請求書が届いたら...（SOPテキストを貼り付け）
```

Claude が自動でSkillファイルを生成・保存します。

### Step 3: テストとチューニング

生成されたSkillを実際のデータで実行し、
エラーや不具合を `CHANGELOG.md` に記録しながら改善する。

---

## 良いSkillの条件

1. **入力が明確**: `$ARGUMENTS` で何を渡すか定義されている
2. **確認ゲートがある**: 書き込み操作の前に人間の承認を求める
3. **ログを残す**: 処理結果を必ずファイルに記録する
4. **失敗を想定している**: エラー時の動作が明示されている

---

## Skill の育て方

Skill は一度作ったら終わりではない。使うたびに改善する。

```
初版: 基本フローを実装
 ↓
実運用: エラーを CHANGELOG.md に記録
 ↓
改善: エラーパターンをルールに追加
 ↓
成熟: 自動処理率が上がり、人間確認が減る
```

これが「PDCAの高速回転」と「知識の蓄積」の実体。
