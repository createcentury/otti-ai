## なぜこの3層構造か

### Layer 3 が「記憶」を担う理由

LLM はセッションをまたいでコンテキストを保持できない。
CLAUDE.md・SKILL.md・CHANGELOG.md をファイルとして保持することで、
「起動するたびにゼロから教える」コストを排除する。

**Progressive Disclosure（段階的開示）**:
SKILL.md はスキルの全詳細をその場では読み込まない。
必要になったタイミングで `skills/` ディレクトリの詳細ファイルを参照する。
これによりトークン消費を最小化しながら、推論精度を維持する。

### Layer 2 が「標準化」を担う理由

MCP (Model Context Protocol) は「AIのためのUSB-C」。
freee、Google Ads、Slack、Notion...すべて同一プロトコルで接続できる。
新しいSaaSを追加する際も、設定ファイルに1エントリ追加するだけ。

### Layer 1 が「実行」を担う理由

Claude Code はローカルのファイルシステムとシェルに直接アクセスできる。
SaaS型AIにデータを送らないため、機密データがクラウドに出ない。
Ralph Loop で自律的なループを維持し、エラーからの自己回復も実現する。

## Human-in-the-Loop の設計

otti-ai は「完全自律」ではなく「準自律」を目指す。

```
読み取り → 分析 → 分類 → [確認ゲート] → 書き込み
                              ↑
                         人間が承認
```

確認ゲートを設けることで:
1. AIの暴走リスクを物理的にブロック
2. 賠償リスクの低減（人間が最終承認）
3. システムが学習する機会（承認/却下のパターンが蓄積）

## スケーラビリティ設計

一度作ったSkillは横展開できる:

```
税理士A事務所で作った freee-check
  → 税理士B事務所にも適用（CLAUDE.md の仕訳ルールを変えるだけ）
  → 同じ freee を使う別業種にも適用可能

マーケターが作った rsa
  → 別のクライアントにも適用（ブランドボイスガイドを差し替えるだけ）
```

`templates/` → コピーして業種別に特化
`skills/` → 再利用可能なモジュールとして保持
